## はじめに — Slackに月額を払い続けるのか問題

小規模チームでSlackを使っていると、こんな不満が出てきます。

- **無料プランだと90日でメッセージが消える**
- 有料にすると1人あたり月額$8.75〜（5人で年間6万円以上）
- 使わない機能が大量にある
- チャンネルが増えすぎて管理しきれない
- 外部サービスにチームの会話データを預ける不安

「必要な機能だけあればいい。自分たちで作れないか？」

そう思って、Next.js + Supabaseで**自社専用のチームボード**を作りました。結果、月額0円で運用でき、必要な機能だけが揃ったシンプルなツールが完成しました。

## 技術スタック

| 項目 | 選定 | 理由 |
|------|------|------|
| フロントエンド | Next.js 14 (App Router) | SSR + SPAのハイブリッド |
| スタイリング | Tailwind CSS | 高速なUI構築 |
| バックエンド・DB | Supabase (PostgreSQL) | リアルタイム同期が標準搭載 |
| リアルタイム | Supabase Realtime | WebSocket、追加コスト0 |
| ホスティング | Vercel | Next.jsとの相性、無料枠 |
| PWA | Service Worker + manifest.json | アプリ化でアクセスしやすく |

**ランニングコスト：¥0**（Supabase無料枠 + Vercel無料枠で収まる規模）

## 実装した機能

### リアルタイムメッセージング

Supabase Realtimeを使って、メッセージの送信・編集・削除がリアルタイムで全メンバーに反映されます。

```typescript
// Supabase Realtimeでメッセージを購読
supabase
  .channel('messages')
  .on('postgres_changes', {
    event: '*',
    schema: 'public',
    table: 'messages',
  }, (payload) => {
    // 新規メッセージ / 編集 / 削除をリアルタイム反映
    handleRealtimeUpdate(payload)
  })
  .subscribe()
```

WebSocketの接続管理やリトライ処理をSupabaseが全部やってくれるので、自前で実装する必要がありません。

### チャンネル＆ダイレクトメッセージ

- **チャンネル**：プロジェクト別・話題別に作成可能（階層構造対応）
- **DM**：メンバー間の1対1メッセージ
- **ドラッグ&ドロップ**で並べ替え
- **アーカイブ**：終了したプロジェクトのチャンネルを非表示に

### メッセージ機能

Slackで「これは欲しい」と思う機能を厳選して実装しました。

- **@メンション**：入力中に候補が表示される
- **リアクション**：👍❤️😊🎉👀✅の6種類
- **返信（スレッド）**：メッセージに直接返信
- **ピン留め**：重要なメッセージをチャンネル上部に固定
- **ブックマーク**：後で見返したいメッセージを保存
- **Markdown**：太字・コード・引用・リストに対応
- **ファイル添付**：画像はインラインプレビュー

### 未読管理

地味だけど重要な機能です。

- サイドバーに**未読バッジ**を表示
- 未読メッセージに**左ボーダー**でハイライト
- **通知ベル**でメンション・DMをまとめて確認
- **ファビコン・タブタイトル**に未読数を反映

### プッシュ通知

Web Push APIで、ブラウザを閉じていてもメンション・DMの通知を受け取れます。

### PWA対応

`manifest.json` + Service Workerで、スマホのホーム画面からアプリとして起動できます。ネイティブアプリを作る必要がなく、URLを共有するだけでチームに配布できます。

### 管理機能

- **アカウント管理**：追加・編集・削除・パスワードリセット
- **権限管理**：管理者 / メンバーの2段階
- **データエクスポート/インポート**：JSON形式でバックアップ・復元
- **全文検索**：全チャンネル横断でメッセージを検索

## データベース設計

PostgreSQL（Supabase）で9テーブル構成です。

```
accounts          — ユーザーアカウント
channels          — チャンネル・DM
messages          — メッセージ本体
reactions         — リアクション
read_messages     — 既読管理
notifications     — 通知
bookmarks         — ブックマーク
online_status     — オンラインステータス
notification_settings — 通知設定
```

既読管理は `read_messages` テーブルで「アカウント × メッセージ」の組み合わせを記録しています。未読数の計算はSQL側でやるため、フロント側の負荷は最小限です。

## Slackと比べてどうか

### 勝っている点

| 項目 | Slack無料 | 自作ボード |
|------|----------|-----------|
| メッセージ保存 | 90日で消える | **無期限** |
| 月額コスト | 5人で¥7,000〜 | **¥0** |
| 機能の複雑さ | 使わない機能が大量 | **必要最小限** |
| データの所在 | Slack社のサーバー | **自社管理** |
| カスタマイズ | 不可 | **自由自在** |

### 負けている点

- **外部サービス連携**（GitHub, Google Drive等）は未実装
- **検索の精度**はSlackの方が上
- **モバイルアプリ**はPWAなのでネイティブには劣る
- **大規模チーム**（50人以上）での運用は未検証

**5〜10人程度の小規模チームなら、自作ボードで十分**というのが実感です。

## 開発期間と工数

- **初期開発**：約2週間（1人で）
- **主な作業**：UI構築5日、リアルタイム同期3日、未読管理2日、PWA対応1日、テスト3日
- **保守**：月1〜2時間（機能追加・バグ修正）

Supabaseがバックエンドの大部分を担ってくれるので、フロントエンドに集中できたのが大きいです。

## セキュリティ面の考慮

自社ツールとはいえ、チームの会話データを扱うので以下を実装しています。

- **認証**：ログインID + パスワード（ハッシュ化保存）
- **権限分離**：管理者のみアカウント管理・チャンネル削除が可能
- **RLS（Row Level Security）**：Supabase側でテーブルレベルのアクセス制御
- **HTTPS**：Vercelが自動で提供
- **アクセス制限**：チームメンバーのみにアカウントを発行

## まとめ — 「作った方が早い」ケースがある

外部SaaSを使うか自作するかの判断基準は：

- **チームが5〜10人以下** → 自作の方がコスパ良い
- **必要な機能が限定的** → 自作の方がシンプル
- **データを自社管理したい** → 自作一択
- **外部連携が多い** → Slack/Teamsの方が良い

Next.js + Supabaseの組み合わせなら、リアルタイム通信・認証・DBがほぼ追加コスト0で使えます。「Slackに毎月払うのはもったいないな」と思ったら、週末に試しに作ってみてください。
