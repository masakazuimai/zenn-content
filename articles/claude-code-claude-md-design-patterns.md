## はじめに

Claude Code を使っていて、**CLAUDE.md を置いているのに思い通りに動いてくれない**と感じたことはないでしょうか。

「ファイルはある。中身もそれなりに書いた。でも指示を無視される、毎回同じ説明をさせられる」——。これ、CLAUDE.md の"設計"が足りていないことが原因です。

筆者は毎日 Claude Code で開発しています。CLAUDE.md を何十回と書き直す中で見えてきたのは、**CLAUDE.md にも設計パターンがある**ということでした。

本記事では、試行錯誤の末に安定したパターンを7つに整理して紹介します。

## 前提：CLAUDE.md の読み込み階層

パターンの前に、CLAUDE.md の階層構造を押さえておきます。Claude Code は以下の順で CLAUDE.md を読み込み、すべてをコンテキストに含めます。

| 階層 | パス | スコープ |
|---|---|---|
| ユーザー | `~/.claude/CLAUDE.md` | 全プロジェクト共通 |
| プロジェクト | `プロジェクトルート/CLAUDE.md` | そのリポジトリ全体 |
| サブディレクトリ | `任意のディレクトリ/CLAUDE.md` | 配下のファイル操作時 |

さらに `~/.claude/rules/*.md` にルールファイルを分割して配置することもできます。

この階層を意識するかどうかで、CLAUDE.md の効き方がまったく変わります。

## パターン 1：グローバルは「人格」、プロジェクトは「仕事」で分ける

最初に陥りがちな罠は、**すべてをプロジェクトの CLAUDE.md に詰め込む**ことです。

技術スタック、コーディングスタイル、言語設定、プロジェクト固有のルール…。これらを1ファイルにまとめると、あっという間に肥大化して「読まれているのかどうかもわからない」状態になります。

**解決策：グローバルとプロジェクトで責務を分離する。**

```markdown
<!-- ~/.claude/CLAUDE.md — 全プロジェクト共通 -->
# グローバル設定

## 技術スタック（デフォルト）
- Next.js（App Router）
- TypeScript
- Tailwind CSS

## 言語設定
- コメントとコミットメッセージは日本語
- 変数名・関数名は英語
```

```markdown
<!-- プロジェクトルート/CLAUDE.md — プロジェクト固有 -->
# プロジェクト概要
このリポジトリはSEO診断ツールのフロントエンド。

## アーキテクチャ
- /app — App Routerのルーティング
- /features — 機能ごとのコンポーネント・ロジック

## 依存関係の注意
- Rechartsを使用（D3.jsは使わない）
- Zustandで状態管理（useContextは避ける）
```

グローバルに書くのは「この開発者がどのプロジェクトでも守りたいこと」。プロジェクトに書くのは「このリポジトリ固有の文脈」。この分離だけで、指示の衝突が大幅に減ります。

## パターン 2：rules/ でドメイン別にファイルを分割する

CLAUDE.md が長くなってきたら、`~/.claude/rules/` ディレクトリにルールを分割します。

```
~/.claude/rules/
├── coding-style.md    # コーディング規約
├── dashboard-design.md  # ダッシュボードUI設計ルール
└── security.md        # セキュリティポリシー
```

**なぜ分割するのか？** 理由は2つあります。

**1. メンテナンス性。** 1ファイル800行のCLAUDE.mdと、200行×4ファイルのrulesでは、後者の方が圧倒的に探しやすく更新しやすい。これはソースコードと同じです。

**2. 再利用性。** coding-style.md は全プロジェクトで使い回せます。プロジェクト固有のルールだけをプロジェクト側のCLAUDE.mdに書けば、DRYになります。

## パターン 3：「やること」より「やらないこと」を書く

CLAUDE.md でありがちなのが、**やってほしいことだけを延々と書く**パターンです。

```markdown
<!-- 効きが弱い例 -->
## ルール
- TypeScriptを使ってください
- エラーハンドリングを適切にしてください
- テストを書いてください
```

これらは Claude Code がデフォルトでやろうとすることです。わざわざ書いても効果は薄い。

**本当に効くのは「やらないこと」の明示です。**

```markdown
## Immutability（CRITICAL）

ALWAYS create new objects, NEVER mutate:

// WRONG: Mutation
function updateUser(user, name) {
  user.name = name  // MUTATION!
  return user
}

// CORRECT: Immutability
function updateUser(user, name) {
  return { ...user, name }
}
```

```markdown
## 禁止事項
- 12px・14pxのフォントサイズは使用しない（最小16px）
- console.logを残さない
- useContextで状態管理しない（Zustandを使う）
- D3.jsを直接使わない（Rechartsを経由する）
```

**「やること」は方針、「やらないこと」はガードレール。** Claude Code の出力精度を上げるのは、ガードレールの方です。

## パターン 4：WRONG / CORRECT のコード例を添える

ルールを文章で書いただけでは、解釈のブレが起きます。**コード例で「悪い例」と「良い例」を並べる**のが最も確実です。

```markdown
## エラーハンドリング

ALWAYS handle errors comprehensively:

try {
  const result = await riskyOperation()
  return result
} catch (error) {
  console.error('Operation failed:', error)
  throw new Error('Detailed user-friendly message')
}
```

これは人間の新人エンジニアに教えるときと同じです。「エラーハンドリングをちゃんとやれ」と言うより、「こう書け」と見せた方が伝わる。Claude Code も同じです。

特に効果が高い場面：
- **イミュータビリティ**（ミューテーションのパターンは無限にある）
- **命名規則**（具体例がないと一貫性が保てない）
- **ファイル構成**（ディレクトリ構造のサンプルを見せる）

## パターン 5：数値で定義できるものは数値にする

「小さいファイルを心がける」ではなく、「800行以下」。「関数は短く」ではなく、「50行以下」。

```markdown
## コード品質チェックリスト

- [ ] 関数は50行以内
- [ ] ファイルは800行以内
- [ ] ネストは4階層以内
- [ ] フォントサイズは16px以上
```

曖昧な指示はClaude Codeに判断を委ねることになり、**あなたの基準とズレた判断をされる原因**になります。数値化できるルールはすべて数値で書きましょう。

## パターン 6：プロジェクト CLAUDE.md に「なぜ」を書く

技術的な制約だけでなく、**プロジェクトの文脈**を書くとClaude Codeの判断精度が上がります。

```markdown
<!-- 効きが弱い -->
## 注意事項
- ビルド時間を短くすること
```

```markdown
<!-- 効きが強い -->
## ビルド最適化（重要）
Vercelの無料枠でBuild Minutesが90%に達している。
まとめpush・キャッシュ活用で対策中。
不要なビルドトリガーを増やす変更は避けること。
```

「なぜ」があると、Claude Code はルールの趣旨を理解して**未知の状況でも適切な判断**ができるようになります。「ビルド時間を短く」だけでは、どの程度の最適化が必要かの判断ができません。

## パターン 7：CLAUDE.md を「生きたドキュメント」にする

CLAUDE.md は一度書いて終わりではありません。**Claude Codeとの対話で見つかった問題をフィードバックし続ける**のが重要です。

実運用でのサイクル：

1. Claude Code が期待と違う出力をする
2. 原因を特定する（ルールの不足？曖昧さ？衝突？）
3. CLAUDE.md を修正する
4. 次回から同じ問題が起きなくなる

たとえば、筆者のケースでは：

- Sonnet に委譲したら品質が落ちた → 「迷ったらOpusに倒す」をルール化
- ダッシュボードのフォントが12pxで生成された → 「最小16px」を明記
- 重いアニメーションでメインスレッドが詰まった → 「OffscreenCanvas + Web Worker優先」を追加

**CLAUDE.md はコードと同じく、リファクタリングの対象です。**

## 設計パターンまとめ

| # | パターン | 核心 |
|---|---|---|
| 1 | グローバル/プロジェクト分離 | 責務を分ける |
| 2 | rules/ で分割 | メンテ性と再利用性 |
| 3 | 「やらないこと」を書く | ガードレールが効く |
| 4 | WRONG/CORRECTコード例 | 曖昧さをゼロにする |
| 5 | 数値で定義する | 判断のブレを防ぐ |
| 6 | 「なぜ」を書く | 未知の判断に強くなる |
| 7 | 生きたドキュメント化 | 継続的改善 |

## おわりに

CLAUDE.md は Claude Code の「設定ファイル」ではなく、**あなたの開発哲学をコードとして表現する場所**です。

単にルールを書き並べるのではなく、「何を禁止するか」「なぜそうするか」「どう書くか」を設計する。この設計の質が、Claude Code の出力精度を決定的に左右します。

CLAUDE.md を雑に書くということは、Claude Code に「適当に判断してくれ」と言っているのと同じです。逆に、ここに時間をかけることで、**毎回の対話で説明し直すコストがゼロになる**。投資対効果は非常に高い。

ぜひ自分のプロジェクトのCLAUDE.mdを見直してみてください。
