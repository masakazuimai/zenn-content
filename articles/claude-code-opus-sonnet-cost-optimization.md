## はじめに

Claude Code を毎日使っていると、必ずぶつかるのが **API コスト問題** です。

特に Opus は Sonnet の約 5 倍の単価。そこで自然に出てくる発想が「**簡単な作業は Sonnet に切り替えればいいじゃないか**」というもの。

でも、実際に数ヶ月運用してみてわかったのは、**安易に Sonnet へ切り替えると総コストが逆に高くなる**という逆説でした。

本記事では、実運用で見えてきた「Opus × Sonnet の正しい役割分担」と、なぜ**『迷ったら Opus』が結局いちばんコストを抑えられるのか**を解説します。

## 結論

先に結論だけ書きます。

- **基本は Opus 単独で進める**
- Sonnet への委譲は「**明確・大量・定型**」の 3 条件が揃ったときだけ
- 委譲は Agent tool の `model: "sonnet"` 指定でサブエージェント化する
- 迷ったら Opus に倒す方が、総コストを抑えられる

「Sonnet で十分」という判断は、**短期のトークン単価だけを見た錯覚**です。実際にはやり直し・追加指示・context 汚染で、むしろ高くつきます。

## なぜ「Sonnet 節約」は失敗するのか

### 1. 一撃精度の差がやり直しコストに直結する

Opus は 1 回の指示で期待通りのコードを返す確率が高い。Sonnet は同じ指示でも**外すことが多い**のが現実です。

外すと何が起きるか:

1. ユーザーが再指示を書く(人間の時間コスト)
2. Sonnet が再生成する(トークンコスト)
3. それでもダメで Opus に切り替える(さらにトークンコスト)
4. Opus が最初からやり直す(最初から Opus でよかった)

1 回の指示で終わらないタスクは、**Opus 単独の方がトータルのトークン消費量が少ない**という逆転現象が起きます。

### 2. context 汚染の複利効果

Sonnet に失敗させた分の会話履歴は、後続のすべてのターンで context に乗り続けます。

- 失敗 → 指摘 → 再実装 → 指摘 → ...

このやりとりが 10 ターン続くと、**後半のターンは毎回ムダに長い context を読み込む**ことになり、気付かないうちに Opus 単独より高額になっていることがあります。

### 3. 判断タスクに Sonnet は向かない

設計判断・リファクタ方針・コードレビュー・仕様の読み解きといった「**考える**」タスクは、Sonnet だと**浅い答え**が返ってきやすい。

こういうタスクは Opus にやらせる方が、1 発で深い回答が得られて総コストを抑えられます。

## それでも Sonnet 委譲が効く 3 条件

一方で、Sonnet 委譲が明確に黒字になるパターンもあります。以下の **3 条件が全て揃ったとき**だけです。

### 条件 1: 指示が明確

- 変更するファイルパスが特定済み
- 変更内容が具体的(追加・削除・置換のどれか明確)
- 完了条件が定義されている(「X を Y にする」レベルで言語化できる)

### 条件 2: 出力量が大きい

- 100 行以上のコード生成
- 複数ファイルの機械的な置換
- 定型テンプレートに沿った実装

出力が小さい(数行の編集)と、**委譲プロンプトと結果受け取りのオーバーヘッドの方が重く**なり、委譲の意味がなくなります。

### 条件 3: 判断が不要

- 「設計をどうするか」を Sonnet に任せない
- 「この実装で良いか」の判断を Sonnet にさせない
- 仕様のグレーゾーンが無い

判断要素が 1 つでも混ざると、Sonnet は外します。**判断は Opus、実装は Sonnet**、と完全に切り分けるのがコツです。

## 実装方法: Agent tool で model 指定

Claude Code では、Agent tool 呼び出し時に `model` パラメータでサブエージェントのモデルを指定できます。

```typescript
// Opus(メインセッション)が判断してから、Sonnet に委譲
Agent({
  description: "定型 CRUD コンポーネントの実装",
  subagent_type: "general-purpose",
  model: "sonnet",
  prompt: `
    src/components/UserList.tsx を新規作成してください。

    【要件】
    - users: User[] を props で受け取る
    - 各ユーザーを Card コンポーネントで表示
    - Card は src/components/ui/Card.tsx に既存
    - TailwindCSS でグリッド(3列)レイアウト
    - 型定義は src/types/user.ts を参照

    【完了条件】
    - TypeScript エラーなし
    - ESLint 警告なし
    - ファイルが 100 行以内
  `
})
```

このとき Opus 側(メイン)は、プロンプトの**読解・設計・レビューだけを担当**します。コードの文字列を出力するトークンは Sonnet 側で消費されるので、Opus のトークン消費を大幅に抑えられます。

## 委譲プロンプトの書き方(失敗しない 3 つのコツ)

Sonnet サブエージェントは**メイン会話の context を共有しません**。だからプロンプトは自己完結させる必要があります。

### コツ 1: ファイルパスを全て明記する

- 悪い例: 「先ほどのコンポーネントを修正してください」
- 良い例: 「`src/components/Header.tsx` の 42 行目の className を `flex` から `grid` に変更」

Sonnet は「先ほど」を知りません。

### コツ 2: 完了条件を言語化する

- 悪い例: 「良い感じに実装してください」
- 良い例: 「TypeScript エラー 0、ESLint 警告 0、テスト全通過」

曖昧な完了条件は Sonnet の得意分野ではありません。

### コツ 3: 判断ポイントは事前に決めておく

- 悪い例: 「適切な状態管理ライブラリを選んでください」
- 良い例: 「Zustand を使ってください(既に package.json にあります)」

選択肢を残すと外します。**判断は Opus 側で終わらせてから**渡します。

## 実際にどれくらいコストが下がるか

個人の実運用ベースの体感値ですが、目安として整理するとこうなります。

| タスクの種類 | 推奨モデル | 理由 |
| --- | --- | --- |
| 設計判断・レビュー・デバッグ | Opus 単独 | 判断精度が総コストに直結 |
| 数行〜数十行の編集 | Opus 単独 | 委譲オーバーヘッドが重い |
| 100 行以上の定型実装 | Opus + Sonnet 委譲 | 30〜50% 削減できる場合あり |
| 複数ファイルの機械的置換 | Opus + Sonnet 委譲 | 明確に黒字 |

**小〜中規模のタスクは Opus 単独、大規模定型タスクだけ委譲**というのが現時点の最適解です。

## まとめ

1. **基本は Opus 単独**。「Sonnet で十分」は罠
2. Sonnet 委譲は「**明確・大量・定型**」の 3 条件が揃ったときだけ
3. 委譲するときは Agent tool で `model: "sonnet"` を指定
4. プロンプトは**自己完結**させる(ファイルパス・完了条件・判断結果を全て渡す)
5. 迷ったら Opus 側に倒す方が、**総コスト**を抑えられる

トークン単価だけを見て Sonnet に切り替えると、やり直しと context 汚染で逆に高くつきます。**一撃精度と総トークン量の両方で評価する**のが、Claude Code の本当にコストを抑えた使い方です。
